plain text는 문자 인코딩을 통해 올바르게 해석될 수 있는 데이터 형식을 말한다. 하지만 정말 우연히 어떤 이진 파일의 숫자가 `01100101011010110110000101100011`과 같이 저장되어, ASCII 문자 인코딩을 통해 `cake`라고 읽을 수 있게 되더라도, 그 의도가 문자 인코딩을 대상으로 한 것이 아니기 때문에 해당 파일은 plain text라고 말할 수 없다.

좀더 간단하게 설명하자면, 메모장으로 열었을때, `꽭뽥뚫$@!`과 같이 나오면 그건 plain text가 아닐 가능성이 높다.

이 글에서는 plain text의 강력함에 대해 얘기해 보려고 한다.

## 매우 많이 쓰인다

plain text가 강력한 이유는 매우 많이 쓰인다는 점에 있다. `utf-8` 문자 인코딩은 어디에나 쓰인다. 이는 가장 큰 장점이다.

- plain text를 지원하는 소프트웨어는 매우 많다. 하지만, `.doc` 같은 파일 들은 특정 소프트웨어에서만 온전히 읽을 수 있다.
- 많은 도구들(예: `grep`, `sed`, `awk`, `find`)이 plain text를 기반으로 동작한다.

그렇다면 이미지 파일(예를 들어 `png`)이 많이 쓰인다면, plain text 만큼 강력할 수 있을까? 그렇다고 생각하는 사람은 거의 없을 것이다. 분명 다른 형식과 plain text에는 구조적인 차이점이 존재한다. 이제 이 차이점에 대해 알아보자.

## 형식을 위한 형식

이미지 파일은 특정 목적을 위한 형식을 가지고 있다. 하지만 plain text 파일은 그렇지 않다. plain text는 그 자체로 의미를 가지지 않으며, 의미를 부여하기 위해 그 위에 또 다른 형식(예: HTML, YAML, Javascript)을 위에 올리고 그 의미를 정의한다.

그렇다면, "굳이 plain text 위에 새로운 형식을 올리지 말고, 그냥 이미지 파일 마냥 직접적으로 0과 1로 표현해도 되는거 아니냐"라고 생각할 수 도 있다. 하지만, 앞서 말했듯이, plain text라는 공통된 기반 위에 또 다른 형식을 올린 것이기 때문에, 설령 그것이 HTML 파일이더라도, plain text로 처리할 수 있는 것이다. 즉, HTML 파서가 아니라, `grep`(문자열 검색기)에도 HTML 파일을 입력으로 넣을 수 있다는 것이다. 또한, 대부분의 코드 편집기가 plain text를 지원하기 때문에, 코드 편집기로 HTML도 수정할 수 있고, 그 외에 C, Java, Javascript 등을 수정할 수 있는 것이다. 만약 사용자가 plain text를 기반으로 하지 않는 새로운 프로그래밍 언어를 만들었다고 해보자. 그렇다면 그 언어는 그 언어만을 위한 새로운 코드 편집기를 필요로 하게된다.

### 왜 plain text인가?

그렇다면 plain text가 형식을 위한 형식으로써 적합한 이유가 뭘까?

일단 형식이란 것은 이것이 옳게된 데이터이며 저건 그렇지 않다는 것을 가르는 기준이다. 프로그래밍 언어의 문법 같은 것이라고 생각하면 된다.

plain text는 문자의 나열이며, 문자들은 순서, 즉 위치정보를 가진다. 문자라는 단위를 문자간의 위치정보, 즉 맥락을 이용하여 조합할 수 있으며, 이 조합은 옳게 되었고, 저 조합은 옳지 않다라는 것을 정의할 수 있다. plain text는 단위의 조합에 탁월하며, 어떤 조합이 유효한 조합인지 정의함으로써, plain text라는 형식 위에 또 다른 형식을 올릴 수 있다.

또한 plain text는 엔디언에 의존하지 않기 때문에, 엔디언 독립적으로 데이터 저장이 가능하다.

혹자는 이렇게 생각할 수 있다. 문자말고 비트를 단위로 써도 단위 조합의 이점을 얻을 수 있지 않나? 하지만, 문자에서 오는 이점이 있다. 그건 다음 섹션인 "코드 포인트"에서 알아보자.

## 코드 포인트(code point)

코드 포인트란 문자를 나타내기 위해 할당된 숫자이다. 문자는 사람이 인식할 수 있는 기호 또는 상징(symbol)이다. 코드 포인트 97은 전 지구적으로 약속된 `a`라는 개념을 가리키는 숫자인 것이다.

분명 이는 plain text의 중요한 특징이다. plain text를 기반으로 하는 여러 형식이 같은 의미에 같은 표현 방식을 쓰도록 어느정도 강제하는 효과가 있기 때문이다. 여기서 짚고 넘어가야 하는 것이 있는데, 같은 표현 방식이라는 것은 대개 사람이 이해하기 쉬운 문자의 조합으로 되어 있다는 것이다. 예를 들어서 사과라는 의미를 표현하기 위해 `apple`이라는 문자의 조합을 쓰는 것이다. HTML 파일에서도, Javascript 파일에서도, Markdown 파일에서도 똑같이 grep을 사용하여 사과라는 의미를 가진 문자의 나열 `apple`을 검색할 수 있는 이유는 이때문이다. 이는 그저 plain text 기반의 여러 형식을 가진 가지각색의 파일들을 `cat` 명령으로 파일을 읽기만 해도 대충 어떤 내용인지 인지할 수 있다는 의미이기도 하다.

또한 사람이 인지하는 기호와 실질적으로 저장되는 숫자가 매핑되어 있기 때문에, 키보드로 손쉬운 입력이 가능하다.

## 언어

plain text를 기반으로하는 형식과 의미의 조합, 즉 일명 plain text를 기반으로하는 언어들은 보통 다음과 같은 특징을 갖는다.

### 자기 기술적(self-describing)

많은 plain text 기반의 형식은 자기 기술적이다. 반면에 이진 파일은 그렇지 않다. 예를 들어서 파일의 `00 00 00 1A`이라는 4 바이트가 나이인지 만두 개수인지 알기 위해서는 해당 이진 파일에 대한 정보가 필요하다. 심지어 숫자 26을 나타내는 것인지도 분명하지 않다. 사과를 나타내는 데이터일 수도 있다는 말이다. 그에 반해 물론 plain text도 해당 파일이 `utf-8` 인코딩을 사용한다는것은 알아야 하긴 하지만, 해당 인코딩으로 데이터를 읽는 것만으로도 바이트열이 무슨 의미인지 알 수 있다. 예를 들어서 JSON의 `"age": 26`을 보자. 26이 나이를 의미한다는 것을 단번에 알 수 있다. 설령 `26`만 덩그러니 있더라도 그게 숫자 26을 뜻할 것이라는 것은 어느정도 유추 가능하다(그 이유는 앞에서 이미 다 설명 되었다).

### 문맥  의존적

많은 이진 파일은 특정한 위치의 바이트가 어떤 의미를 나타내는지 지정되어 있다(예를 들어 9번째 바이트부터, 4바이트는 나이). 그에 반해 대부분의 plain text위에 얹어진 형식(언어)은 문자열에서 서로의 상대적인 위치에 따른 맥락을 가지도록 설계되어 있다(예를 들어`"age": 26` 은 `"age"`와의 관계(맥락)를 통해  `26`이 나이라는 것을 안다). 이러한 특징 때문에 plain text는 파일 중간이 손상되어도, 그 앞뒤의 의미를 어느정도 보존할 수 있다(`utf-8` 같은 인코딩은 문자의 경계 식별이 가능하기 때문에 몇 바이트 없어져도 문자의 식별이 가능하다). 또한 중간에 데이터가 삽입되거나 삭제되어도 전체 구조가 무너지지 않는 확장성을 보인다. 그에 반해 이진 파일은 보통 미래를 위해 예약된 영역을 두는 경향이 있다.

## 사람이 읽을 수 있는

다시 짚어보자면 plain text기반의 여러 형식들은 공통된 문자 아래에서 같은 의미에 비슷한 표현 방식을 공유하며, 대부분 사람이 이해하기 쉬운 문자의 조합으로 되어 있다. 또한 자기 기술적이며, 문맥 의존적인 특징은 사람의 언어와 비슷하기 때문에 plain text를 사람이 읽기 쉽게 한다.

사실 모든 plain text가 사람이 읽기에 좋은 것은 아니다. 엑셀 파일(`xlsx`)은 사실 내부적으로 XML로 되어 있지만, 사용자가 다루기 위해서는 엑셀 파일을 위한 소프트웨어를 필요로 한다. 복잡도가 사람의 인지의 한계를 넘어섰기 때문이다.

## LLM과의 궁합

요즘의 LLM(예: ChatGPT, Gemini)은 토큰의 나열을 입력으로 받고 출력한다. 대충 문자열을 입력받고 출력한다고 이해하면 된다. LLM은 문자열의 맥락을 파악하여, 그 다음에 나올 문자를 예측하는 모델이다. 앞서 말했듯이 plain text 데이터는 일반적으로 맥락을 가지며, 문자의 조합(LLM에서는 이를 토큰 임베딩을 통해 의미를 표현한다)이 의미를 가지는 특징이 있기 때문에 LLM과의 궁합이 매우 좋다.

## 시퀀스 데이터

plain text는 문자의 나열이며, 따라서 plain text는 프로그래밍에서 시퀀스 데이터 구조로 다루기 쉽다. 문자열을 지원하지 않는 프로그래밍 언어는 찾아보기 어렵다. 이는 plain text의 또다른 장점이다.

## 끝내며...

Unix 계열 운영체제는 plain text를 적극적으로 사용한다. 일단 shell부터가 모든 데이터가 문자열이며, 여러가지 도구들은 파이프를 통해 문자열을 주고받는다. 또한 시스템 정보를 파일을 통해 plain text로 제공한다. plain text를 적극적으로 사용하는 또 다른 곳은 웹이다. HTTP는 plain text이며, HTML, CSS, Javascript 또한 그렇다.

또한 Git을 빼놓을 수 없는데, Git은 diff라는 것을 기반으로 하며, diff는 텍스트의 각 줄을 비교 단위로 사용한다. 이는 각 줄을 비교 단위로 사용하는 것이 적합한 비교 방법일 것이라는 가정 하에 작동하며, 대부분의 plain text 기반의 언어들에서 유효한 방법이기 때문에 Git은 많은 형식을 똑같은 비교 방법으로 성공적으로 처리할 수 있다. 하지만, 이진 파일의 경우에는 적합한 방법이 제각기 다를 수 있으며, Git은 이진 파일에 대해 파일이 바뀌었다는 것 외에는 구체적인 비교정보를 제공하지 않는다.

plain text는 상호운용성의 정점이다. 상호운용성이란 "서로 다른 시스템들이 별도의 특별한 노력 없이도 정보를 교환하고 사용할 수 있는 능력"을 말한다. plain text는 기반 형식으로써 시스템들이 비트 뭉치를 똑같은 문자로 해석할 수 있게 해주며, plain text를 기반으로 하는 언어들의 구조적 특징은, `grep`이나 `sed` 같은 도구들이 언어를 몰라도 "텍스트 패턴"이라는 공통 인터페이스를 통해 유의미한 처리를 할 수 있게 해준다. plain text는 심지어 사람과, 기계사이에서도 상호운용성을 제공한다. 이것이 plain text가 강력한 이유이다.
